# IFC-V1-026 Device Cookie Authentication Context

Date: 2026-07-11

Status: complete. Implementation: `9b88d6d`.

## Scope

Implement the selected Fastify request authentication context after the completed raw-socket transport/Host/Origin trust gate. This leaf owns strict inbound device-cookie parsing, explicit loopback local-admin provenance, lazy once-per-request bearer authentication through the monotonic auth-device repository, one frozen non-secret context, and reusable enforcement for the selected route-manifest authentication mechanisms. It does not issue or clear cookies, rotate/verify CSRF, rate-limit, lock, audit, register selected API routes, authorize a specific target, recheck long-lived SSE authority, or implement UI state.

## Pre-Implementation Gaps

- The selected Fastify app installs transport/Host/Origin trust but has no device-cookie parser, authentication input, per-request auth state, auth diagnostics, or route pre-handler enforcement.
- Historical `host-service`/`security-routes` code splits cookie strings manually, decodes noncanonical values, accepts the first duplicate, infers local admin from weak browser-header absence, catches every auth failure as untrusted, and can authenticate/touch twice for one status request. It is not a selected implementation base.
- `AuthDeviceRepository.authenticateDeviceToken` now gives strict immediate monotonic authority and last-used behavior, but no server boundary sanitizes its corrupt/native errors or maps invalid, expired, revoked, conflict, and unavailable states without exposing the bearer.
- Eager global authentication would touch SQLite for liveness, static assets, and routes that do not consume identity; parallel browser assets would also create unnecessary monotonic-write conflicts.
- The selected 43-character base64url device secret generated by pairing storage is not exported as a shared contract, so HTTP validation would otherwise duplicate or weaken its syntax.
- Route-manifest auth mechanisms exist, but there is no reusable gate proving that unpaired LAN reads, invalid supplied credentials, read-only writes, or nonlocal local-admin claims stop before a protected handler.

## Implementation Result

- Added the shared exact 43-character selected device-secret contract and one strict request-authentication context contract with canonical origin/transport, state/authority, lifecycle, permission, and CSRF-generation coherence.
- Added exact direct `cookie@1.1.1` parsing with identity decoding, raw-header and target-duplicate rejection, case-sensitive fixed cookie name, strict selected-token syntax, and no ambient local-admin fallback when any Cookie header is present.
- Added a branded mandatory authentication policy plus root post-trust intake. Private request-keyed pending authority resolves lazily once, clears deterministically after resolution/response/abort, validates the full repository result, and caches one frozen non-secret context or one sanitized failure.
- Added explicit local-admin, unpaired, invalid, expired, revoked, and paired-device states with coherent read/write permissions; manifest-aligned auth helpers; separate write-permission enforcement; saturating count-only diagnostics; and fixed cause-free 401/403/409/500 behavior.
- Made the app factory and selected lifecycle require a valid context-derived auth policy before route construction/app creation. Existing app/trust/SSE/static/lifecycle/resource fixtures use a reject-on-use policy so unrelated tests prove no hidden auth resolution.
- Reused the shared secret contract in selected pairing generation, removing the duplicate private device-secret regex without changing generated credential behavior.

## Frozen Selected Contract

### Dependency And Cookie Intake

- Add exact direct `cookie@1.1.1`, already present in the reviewed Fastify dependency graph. It is MIT, has zero dependencies, supports Node 18+, and is above the `0.7.0` fix for `CVE-2024-47764`. The newly released 2.x API is not introduced during this security leaf.
- The parser uses the package's structured request-cookie parser with identity decoding. The selected device token remains exactly 43 unpadded base64url characters; quoted, percent-encoded, whitespace-bearing, empty, short, long, Unicode, or alternate textual encodings do not normalize into valid authority.
- Read raw headers after the trust gate. Zero or one raw `Cookie` field is accepted; duplicate fields or duplicate exact `hostdeck_device` assignments are invalid. Cookie names remain case-sensitive. Bounded unrelated cookies may coexist but never become auth metadata.
- No target cookie means unpaired. A malformed/duplicate target means `invalid_device` and invokes no clock or repository. A present cookie header of any kind prevents ambient local-admin fallback.

### Local Admin Provenance

- Local admin exists only when the admitted trust context is `local_non_browser` and the request has no `Cookie` header. That trust classification already requires a loopback peer, an unsafe missing-Origin request, no browser fetch metadata/preflight, exact configured Host, allowed transport, and no forwarding headers.
- `same_origin` or `safe_no_origin` traffic without a device cookie is unpaired, never local admin. A supplied invalid, expired, revoked, read-only, or valid device credential is handled as that device state and cannot fall back to local admin.
- Loopback read policy is distinct from local admin. The selected `loopback_or_device_cookie` mechanism may admit an unpaired request only when `network_mode` is `loopback`; LAN always requires a valid paired device before protected data.

### Lazy Once-Per-Request Authentication

- A mandatory branded authentication policy provides only the synchronous bearer-authentication port and clock. The app factory installs cookie intake after request resource admission and request trust, before route hooks. The selected lifecycle requires a context-derived policy before app creation/readiness/listen.
- Intake stores pending request state in a private `WeakMap`; no raw token is attached to Fastify request objects, logs, diagnostics, contexts, errors, or replies.
- Authentication resolves lazily from a protected route pre-handler/helper. A public, not-found, or static request that never resolves auth performs no clock call, hash, SQLite read, or last-used write.
- First resolution snapshots the clock once and invokes `authenticateDeviceToken` once. It validates the returned full internal result, projects and freezes the selected context, clears the pending raw token, and caches the context for every later consumer in that request.
- Repository monotonic behavior is preserved without retry: equal time is a no-op success, a regressing/concurrent observation is a visible `operation_conflict`, and storage/corrupt/unexpected failure is a fixed cause-free `storage_error`. No failed operation returns paired authority.

### Exact Non-Secret Context

Every resolved context contains only:

- `state`: `local_admin`, `unpaired`, `invalid_device`, `expired_device`, `revoked_device`, or `paired_device`;
- admitted `configured_origin`, `network_mode`, `origin_kind`, and actual `transport`;
- nullable `device_id`, `permission`, `csrf_generation`, `last_used_at`, and `expires_at`.

`local_admin` has permission `local_admin` and no device fields. `paired_device` has one selected device id, `read` or `write` permission, positive safe CSRF generation, canonical non-null last-used time, and nullable canonical expiry. Unpaired/invalid/expired/revoked contexts expose no device identity or metadata. The outer context is exact and frozen; no bearer/CSRF hash, raw token, label, revocation timestamp, database row, Cookie header, request target, or unrelated state is included.

### Manifest-Aligned Enforcement

- One reusable pre-handler helper resolves and enforces `optional_device_cookie`, `loopback_or_device_cookie`, `device_cookie`, `local_admin`, or `local_admin_or_device_cookie`. `none` and `pairing_code` remain outside this leaf.
- Invalid/expired/revoked supplied credentials never fall back to loopback read. Missing LAN credentials reject before protected handlers. Local-admin-only policy accepts only the exact local-admin context. Device-cookie policy accepts only paired state.
- A separate write-permission assertion accepts local admin or paired `write` and rejects paired `read` as `read_only`. CSRF, lock, target, rate, and audit checks remain mandatory downstream and are not implied by this assertion.
- Credential rejection is fixed, bounded, cause-free, and non-reflective: absent/invalid/expired/revoked states remain distinguishable without returning token or device identity.

## Remaining Gaps

None within this leaf. Cookie issuance, CSRF authorization, concrete protected routes, target/write gates, security audit orchestration, and long-lived authority rechecks remain with the downstream owners below.

## Reuse Assessment

Reuse the strict request-trust context and monotonic `AuthDeviceRepository.authenticateDeviceToken` transaction. Add direct `cookie@1.1.1` for RFC request-cookie parsing, wrapped only for raw-header multiplicity, target duplication, identity decoding, and selected-token validation that the generic parser cannot own. Do not reuse the historical custom listener parser, add signed-cookie state, or add `@fastify/cookie` merely to expose a mutable request-wide cookie bag.
